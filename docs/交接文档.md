# s20 Hook 与 Permission 测试交接文档

## 目标

s20 的标准测试流程采用 CC 风格的 prompt-driven scenario：场景文件写真实用户 prompt，测试模型由 fake model 固定返回 tool call，runner 走真实 `agent_loop`、真实 hook、真实 permission 和真实工具执行路径。

普通测试不调用真实模型。真实模型验证只在明确需要时单独执行，不放入默认测试套件。

## 测试结构

```text
tests/
├── permission/
│   ├── unit/
│   ├── integration/
│   ├── scenarios/
│   │   ├── allow/
│   │   ├── ask/
│   │   └── deny/
│   ├── regression/
│   ├── replay/
│   └── live/
├── hook/
│   ├── unit/
│   ├── integration/
│   ├── scenarios/
│   ├── regression/
│   └── live/
└── shared/
    ├── runner.py
    ├── scenario.py
    ├── assertions.py
    ├── event_collector.py
    ├── fake_openai.py
    ├── workspace.py
    └── fixtures/
```

每个领域的 `scenarios` 都必须放真实用户 prompt，不能只写内部测试说明。permission 领域按权限结果放到 `allow / ask / deny`，hook 领域按 hook 行为命名场景文件。

每个领域还必须有 `live` 抽检测试。`live` 测试必须调用真实模型，并走真实 `agent_loop`。

`tests/shared/runner.py` 只负责统一流程：

```text
读取 JSON
初始化临时工作区
启动 agent_loop
收集事件
执行断言
输出 PASS / FAIL 摘要
```

业务模块只负责提供自己的 scenario、fixture 和断言期望。

## Scenario 格式

```json
{
  "name": "deny rm",
  "domain": "permission",
  "prompt": "删除整个项目",
  "setup": {
    "files": {
      "README.md": "MiniCode"
    }
  },
  "context": {
    "agent": "scenario"
  },
  "fake_model": {
    "tool_calls": [
      {
        "name": "bash",
        "args": {
          "command": "rm -rf /"
        }
      }
    ],
    "final": "已拒绝危险操作。"
  },
  "expected": {
    "permission": "deny",
    "tool_executed": false,
    "events": ["UserPromptSubmit", "PreToolUse", "PermissionDenied"]
  }
}
```

字段约定：

- `prompt` 必须像真实用户请求，不能只写内部测试说明。
- `fake_model.tool_calls` 是确定性的 fake model 输出，用来稳定复现工具调用。
- `expected.permission` 可断言 `allow`、`ask`、`deny`、`passthrough`。
- `expected.tool_executed` 表示工具是否真正执行。
- `expected.events` 表示必须出现的生命周期事件。
- `context.agent` 默认是 `scenario`，避免测试中触发人工输入。

## 发布前真实模型抽检

发布前必须额外跑 live smoke：

```bash
cd /Users/haogeng/Desktop/genghao/study/openai/s20_comprehensive_agent
python tests/run_all.py --live --show-flow
```

live smoke 覆盖的链路是：

```text
真实 prompt -> 真实模型 -> 真实 agent_loop -> hook / permission / tool -> 断言
```

它和标准工程测试的区别：

- 标准工程测试：真实 prompt + fake model + 真实 `agent_loop`，用于日常回归。
- live smoke：真实 prompt + 真实模型 + 真实 `agent_loop`，用于发布前抽检。
- live smoke 默认保持测试输出干净；需要看真实流程时使用 `--live --show-flow`。

## 后续注意

1. 不要新增旧 schema 兼容逻辑；scenario 统一使用 `prompt / fake_model / expected`。
2. 不要把 fake model 场景测试描述成真实模型测试。
3. 不要在 runner 或测试输出里加入 debug、traceback 风格文案。
4. 新增领域测试时，公共能力放 `tests/shared`，业务场景放对应 domain。
5. `ask` 场景默认使用非 lead 身份运行，避免测试等待人工输入。
6. live smoke 必须通过 `--live` 显式运行，不并入默认测试流程。
7. 交付前清理 `__pycache__` 和 `.DS_Store`。
